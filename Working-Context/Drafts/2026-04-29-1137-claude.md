This includes:\n226\t\n227\t- Core project keys: `GEMINI_API_KEY`, `GOOGLE_API_KEY`, etc.\n228\t- Variables matching sensitive patterns: `*TOKEN*`, `*SECRET*`, `*PASSWORD*`,\n229\t  `*KEY*`, `*AUTH*`, `*CREDENTIAL*`.\n230\t- Certificates and private key patterns.\n231\t\n232\t#### Explicit overrides\n233\t\n234\tIf an environment variable must be passed to an MCP server, you must explicitly\n235\tstate it in the `env` property of the server configuration in `settings.json`.\n236\tExplicitly defined variables (including those from extensions) are trusted and\n237\tare **not** subjected to the automatic redaction process.\n238\t\n239\tThis follows the security principle that if a variable is explicitly configured\n240\tby the user for a specific server, it constitutes informed consent to share that\n241\tspecific data with that server.\n242\t\n243\t<!-- prettier-ignore -->\n244\t> [!NOTE]\n245\t> Even when explicitly defined, you should avoid hardcoding secrets.\n246\t> Instead, use environment variable expansion\n247\t> (for example, `"MY_KEY": "$MY_KEY"`) to securely pull the value from your host\n248\t> environment at runtime.\n249\t\n250\t### OAuth support for remote MCP servers\n251\t\n252\tGemini CLI supports OAuth 2.0 authentication for remote MCP servers using SSE or\n253\tHTTP transports. This enables secure access to MCP servers that require\n254\tauthentication.\n255\t\n256\t#### Automatic OAuth discovery\n257\t\n258\tFor servers that support OAuth discovery, you can omit the OAuth configuration\n259\tand let the CLI discover it automatically:\n260\t\n261\t```json\n262\t{\n263\t  "mcpServers": {\n264\t    "discoveredServer": {\n265\t      "url": "https://api.example.com/sse"\n266\t    }\n267\t  }\n268\t}\n269\t```\n270\t\n271\tThe CLI will automatically:\n272\t\n273\t- Detect when a server requires OAuth authentication (401 responses)\n274\t- Discover OAuth endpoints from server metadata\n275\t- Perform dynamic client registration if supported\n276\t- Handle the OAuth flow and token management\n277\t\n278\t#### Authentication flow\n279\t\n280\tWhen connecting to an OAuth-enabled server:\n281\t\n282\t1. **Initial connection attempt** fails with 401 Unauthorized\n283\t2. **OAuth discovery** finds authorization and token endpoints\n284\t3. **Browser opens** for user authentication (requires local browser access)\n285\t4. **Authorization code** is exchanged for access tokens\n286\t5. **Tokens are stored** securely for future use\n287\t6. **Connection retry** succeeds with valid tokens\n288\t\n289\t#### Browser redirect requirements\n290\t\n291\t<!-- prettier-ignore -->\n292\t> [!IMPORTANT]\n293\t> OAuth authentication requires that your local machine can:\n294\t>\n295\t> - Open a web browser for authentication\n296\t> - Receive redirects on `http://localhost:<random-port>/oauth/callback` (or a specific port if configured via `redirectUri`)\n297\t\n298\tThis feature will not work in:\n299\t\n300\t- Headless environments without browser access\n301\t- Remote SSH sessions without X11 forwarding\n302\t- Containerized environments without browser support\n303\t\n304\t#### Managing OAuth authentication\n305\t\n306\tUse the `/mcp auth` command to manage OAuth authentication:\n307\t\n308\t```bash\n309\t# List servers requiring authentication\n310\t/mcp auth\n311\t\n312\t# Authenticate with a specific server\n313\t/mcp auth serverName\n314\t\n315\t# Re-authenticate if tokens expire\n316\t/mcp auth serverName\n317\t```\n318\t\n319\t#### OAuth configuration properties\n320\t\n321\t- **`enabled`** (boolean): Enable OAuth for this server\n322\t- **`clientId`** (string): OAuth client identifier (optional with dynamic\n323\t  registration)\n324\t- **`clientSecret`** (string): OAuth client secret (optional for public clients)\n325\t- **`authorizationUrl`** (string): OAuth authorization endpoint (auto-discovered\n326\t  if omitted)\n327\t- **`tokenUrl`** (string): OAuth token endpoint (auto-discovered if omitted)\n328\t- **`scopes`** (string[]): Required OAuth scopes\n329\t- **`redirectUri`** (string): Custom redirect URI (defaults to an OS-assigned\n330\t  random port, e.g., `http://localhost:<random-port>/oauth/callback`)\n331\t- **`tokenParamName`** (string): Query parameter name for tokens in SSE URLs\n332\t- **`audiences`** (string[]): Audiences the token is valid for\n333\t\n334\t#### Token management\n335\t\n336\tOAuth tokens are automatically:\n337\t\n338\t- **Stored securely** in `~/.gemini/mcp-oauth-tokens.json`\n339\t- **Refreshed** when expired (if refresh tokens are available)\n340\t- **Validated** before each connection attempt\n341\t- **Cleaned up** when invalid or expired\n342\t\n343\t#### Authentication provider type\n344\t\n345\tYou can specify the authentication provider type using the `authProviderType`\n346\tproperty:\n347\t\n348\t- **`authProviderType`** (string): Specifies the authentication provider. Can be\n349\t  one of the following:\n350\t  - **`dynamic_discovery`** (default): The CLI will automatically discover the\n351\t    OAuth configuration from the server.\n352\t  - **`google_credentials`**: The CLI will use the Google Application Default\n353\t    Credentials (ADC) to authenticate with the server. When using this provider,\n354\t    you must specify the required scopes.\n355\t  - **`service_account_impersonation`**: The CLI will impersonate a Google Cloud\n356\t    Service Account to authenticate with the server. This is useful for\n357\t    accessing IAP-protected services (this was specifically designed for Cloud\n358\t    Run services).\n359\t\n360\t#### Google credentials\n361\t\n362\t```json\n363\t{\n364\t  "mcpServers": {\n365\t    "googleCloudServer": {\n366\t      "httpUrl": "https://my-gcp-service.run.app/mcp",\n367\t      "authProviderType": "google_credentials",\n368\t      "oauth": {\n369\t        "scopes": ["https://www.googleapis.com/auth/userinfo.email"]\n370\t      }\n371\t    }\n372\t  }\n373\t}\n374\t```\n375\t\n376\t#### Service account impersonation\n377\t\n378\tTo authenticate with a server using Service Account Impersonation, you must set\n379\tthe `authProviderType` to `service_account_impersonation` and provide the\n380\tfollowing properties:\n381\t\n382\t- **`targetAudience`** (string): The OAuth Client ID allowlisted on the\n383\t  IAP-protected application you are trying to access.\n384\t- **`targetServiceAccount`** (string): The email address of the Google Cloud\n385\t  Service Account to impersonate.\n386\t\n387\tThe CLI will use your local Application Default Credentials (ADC) to generate an\n388\tOIDC ID token for the specified service account and audience. This token will\n389\tthen be used to authenticate with the MCP server.\n390\t\n391\t#### Setup instructions\n392\t\n393\t1. **[Create](https://cloud.google.com/iap/docs/oauth-client-creation) or use process deep dive\n537\t\n538\tWhen Gemini CLI starts, it performs MCP server discovery through the following\n539\tdetailed process:\n540\t\n541\t### 1. Server iteration and connection\n542\t\n543\tFor each configured server in `mcpServers`:\n544\t\n545\t1. **Status tracking begins:** Server status is set to `CONNECTING`\n546\t2. **Transport selection:** Based on configuration properties:\n547\t   - `httpUrl` → `StreamableHTTPClientTransport`\n548\t   - `url` → `SSEClientTransport`\n549\t   - `command` → `StdioClientTransport`\n550\t3. **Connection establishment:** The MCP client attempts to connect with the\n551\t   configured timeout\n552\t4. **Error handling:** Connection failures are logged and the server status is\n553\t   set to `DISCONNECTED`\n554\t\n555\t### 2. Tool discovery\n556\t\n557\tUpon successful connection:\n558\t\n559\t1. **Tool listing:** The client calls the MCP server\'s tool listing endpoint\n560\t2. **Schema validation:** Each tool\'s function declaration is validated\n561\t3. **Tool filtering:** Tools are filtered based on `includeTools` and\n562\t   `excludeTools` configuration\n563\t4. **Name sanitization:** Tool names are cleaned to meet Gemini API\n564\t   requirements:\n565\t   - Characters other than letters, numbers, underscore (`_`), hyphen (`-`), dot\n566\t     (`.`), and colon (`:`) are replaced with underscores\n567\t   - Names longer than 63 characters are truncated with middle replacement\n568\t     (`...`)\n569\t\n570\t### 3. Tool naming and namespaces\n571\t\n572\tTo prevent collisions across multiple servers or conflicting built-in tools,\n573\tevery discovered MCP tool is assigned a strict namespace.\n574\t\n575\t1. **Automatic FQN:** All MCP tools are unconditionally assigned a fully\n576\t   qualified name (FQN) using the format `mcp_{serverName}_{toolName}`.\n577\t2. **Registry tracking:** The tool registry maintains metadata mappings between\n578\t   these FQNs and their original server identities.\n579\t3. **Overwrites:** If two servers share the exact same alias in your\n580\t   configuration and provide tools with the exact same name, the last registered\n581\t   tool overwrites the previous one.\n582\t4. **Policies:** To configure permissions (like auto-approval or denial) for MCP\n583\t   tools, see\n584\t   [Special syntax for MCP tools](../reference/policy-engine.md#special-syntax-for-mcp-tools)\n585\t   in the Policy Engine documentation.\n586\t\n587\t<!-- prettier-ignore -->\n588\t> [!WARNING]\n589\t> Do not use underscores (`_`) in your MCP server names (for example, use\n590\t> `my-server` rather than `my_server`). The policy parser splits Fully Qualified\n591\t> Names (`mcp_server_tool`) on the _first_ underscore following the `mcp_`\n592\t> prefix. If your server name contains an underscore, the parser will\n593\t> misinterpret the server identity, which can cause wildcard rules and security\n594\t> policies to fail silently.\n595\t\n596\t### 4. Schema processing\n597\t\n598\tTool parameter schemas undergo sanitization for Gemini API compatibility:\n599\t\n600\t- **`$schema` properties** are removed\n601\t- **`additionalProperties`** are stripped\n602\t- **`anyOf` with `default`** have their default values removed (Vertex AI\n603\t  compatibility)\n604\t- **Recursive processing** applies to nested schemas\n605\t\n606\t### 5. Connection management\n607\t\n608\tAfter discovery:\n609\t\n610\t- **Persistent connections:** Servers that successfully register tools maintain\n611\t  their connections\n612\t- **Cleanup:** Servers that provide no usable tools have their connections\n613\t  closed\n614\t- **Status updates:** Final server statuses are set to `CONNECTED` or\n615\t  `DISCONNECTED`\n616\t\n617\t## Tool execution flow\n618\t\n619\tWhen the Gemini model decides to use an MCP tool, the following execution flow\n620\toccurs:\n621\t\n622\t### 1. Tool invocation\n623\t\n624\tThe model generates a `FunctionCall` with:\n625\t\n626\t- **Tool name:** The registered name (potentially prefixed)\n627\t- **Arguments:** JSON object matching the tool\'s parameter schema\n628\t\n629\t### 2. Confirmation process\n630\t\n631\tEach `DiscoveredMCPTool` implements sophisticated confirmation logic:\n632\t\n633\t#### Trust-based bypass\n634\t\n635\t```typescript\n636\tif (this.trust) {\n637\t  return false; // No confirmation needed\n638\t}\n639\t```\n640\t\n641\t#### Dynamic allow-listing\n642\t\n643\tThe system maintains internal allow-lists for:\n644\t\n645\t- **Server-level:** `serverName` → All tools from this server are trusted\n646\t- **Tool-level:** `serverName.toolName` → This specific tool is trusted\n647\t\n648\t#### User choice handling\n649\t\n650\tWhen confirmation is required, users can choose:\n651\t\n652\t- **Proceed once:** Execute this time only\n653\t- **Always allow this tool:** Add to tool-level allow-list\n654\t- **Always allow this server:** Add to server-level allow-list\n655\t- **Cancel:** Abort execution\n656\t\n657\t### 3. Execution\n658\t\n659\tUpon confirmation (or trust bypass):\n660\t\n661\t1. **Parameter preparation:** Arguments are validated against the tool\'s schema\n662\t2. **MCP call:** The underlying `CallableTool` invokes the server with:\n663\t\n664\t   ```typescript\n665\t   const functionCalls = [\n666\t     {\n667\t       name: this.serverToolName, // Original server tool name\n668\t       args: params,\n669\t     },\n670\t   ];\n671\t   ```\n672\t\n673\t3. **Response processing:** Results are formatted for both LLM context and user\n674\t   display\n675\t\n676\t### 4. Response handling\n677\t\n678\tThe execution result contains:\n679\t\n680\t- **`llmContent`:** Raw response parts for the language model\'s context\n681\t- **`returnDisplay`:** Formatted output for user display (often JSON in markdown\n682\t  code blocks)\n683\t\n684\t## How to interact with your MCP server\n685\t\n686\t### Using the `/mcp` command\n687\t\n688\tThe `/mcp` command provides comprehensive information about your MCP server\n689\tsetup:\n690\t\n691\t```bash\n692\t/mcp\n693\t```\n694\t\n695\tThis displays:\n696\t\n697\t- **Server list:** All configured MCP servers\n698\t- **Connection status:** `CONNECTED`, `CONNECTING`, or `DISCONNECTED`\n699\t- **Server details:** Configuration summary (excluding sensitive data)\n700\t- **Available tools:** List of tools from each server with descriptions\n701\t- **Discovery state:** Overall discovery process status\n702\t\n703\t### Example `/mcp` output\n704\t\n705\t```\n706\tMCP Servers Status:\n707\t\n708\t📡 pythonTools (CONNECTED)\n709\t  Command: python -m my_mcp_server --port 8080\n710\t  Working Directory: ./mcp-servers/python\n711\t  Timeout: 15000ms\n712\t  Tools: calculate_sum, file_analyzer, data_processor\n713\t\n714\t🔌 nodeServer (DISCONNECTED)\n715\t  Command: node dist/server.js --verbose\n716\t  Error: Connection refused\n717\t\n718\t🐳 dockerizedServer (CONNECTED)\n719\t  Command: docker run -i --rm -e API_KEY my-mcp-server:latest\n720\t  Tools: mcp_dockerizedServer_docker_deploy, mcp_dockerizedServer_docker_status\n721\t\n722\tDiscovery State: COMPLETED\n723\t```\n724\t\n725\t### Tool usage\n726\t\n727\tOnce discovered, MCP tools are available to the Gemini model like built-in\n728\ttools. The model will automatically:\n729\t\n730\t1. **Select appropriate tools** based on your requests\n731\t2. **Present confirmation dialogs** (unless the server is trusted)\n732\t3. **Execute tools** with proper parameters\n733\t4. **Display results** in a user-friendly format\n734\t\n735\t## Status monitoring and troubleshooting\n736\t\n737\t### Connection states\n738\t\n739\tThe MCP integration tracks several states:\n740\t\n741\t#### Overriding extension configurations\n742\t\n743\tIf an MCP server is provided by an extension (for example, the\n744\t`google-workspace` extension), you can still override its settings in your local\n745\t`settings.json`. Gemini CLI merges your local configuration with the extension\'s\n746\tdefaults:\n747\t\n748\t- **Tool lists:** Tool lists are merged securely to ensure the most restrictive\n749\t  policy wins:\n750\t  - **Exclusions (`excludeTools`):** Arrays are combined (unioned). If either\n751\t    source blocks a tool, it remains disabled.\n752\t  - **Inclusions (`includeTools`):** Arrays are intersected. If both sources\n753\t    provide an allowlist, only tools present in **both** lists are enabled. If\n754\t    only one source provides an allowlist, that list is respected.\n755\t  - **Precedence:** `excludeTools` always takes precedence over `includeTools`.\n756\t\n757\t  This ensures you always have veto power over tools provided by an extension\n758\t  and that an extension cannot re-enable tools you have omitted from your\n759\t  personal allowlist.\n760\t\n761\t- **Environment variables:** The `env` objects are merged. If the same variable\n762\t  is defined in both places, your local value takes precedence.\n763\t- **Scalar properties:** Properties like `command`, `url`, and `timeout` are\n764\t  replaced by your local values if provided.\n765\t\n766\t**Example override:**\n767\t\n768\t```json\n769\t{\n770\t  "mcpServers": {\n771\t    "google-workspace": {\n772\t      "excludeTools": ["gmail.send"]\n773\t    }\n774\t  }\n775\t}\n776\t```\n777\t\n778\t#### Server status (`MCPServerStatus`)\n779\t\n780\t- **`DISCONNECTED`:** Server is not connected or has errors\n781\t- **`CONNECTING`:** Connection attempt in progress\n782\t- **`CONNECTED`:** Server is connected and ready\n783\t\n784\t#### Discovery state (`MCPDiscoveryState`)\n785\t\n786\t- **`NOT_STARTED`:** Discovery hasn\'t begun\n787\t- **`IN_PROGRESS`:** Currently discovering servers\n788\t- **`COMPLETED`:** Discovery finished (with or without errors)\n789\t\n790\t### Common issues and solutions\n791\t\n792\t#### Server won\'t connect\n793\t\n794\t**Symptoms:** Server shows `DISCONNECTED` status\n795\t\n796\t**Troubleshooting:**\n797\t\n798\t1. **Check configuration:** Verify `command`, `args`, and `cwd` are correct\n799\t2. **Test manually:** Run the server command directly to ensure it works\n800\t3. **Check dependencies:** Ensure all required packages are installed\n801\t4. **Review logs:** Look for error messages in the CLI output\n802\t5. **Verify permissions:** Ensure the CLI can execute the server command\n803\t\n804\t#### No tools discovered\n805\t\n806\t**Symptoms:** Server connects but no tools are available\n807\t\n808\t**Troubleshooting:**\n809\t\n810\t1. **Verify tool registration:** Ensure your server actually registers tools\n811\t2. **Check MCP protocol:** Confirm your server implements the MCP tool listing\n812\t   correctly\n813\t3. **Review server logs:** Check stderr output for server-side errors\n814\t4. **Test tool listing:** Manually test your server\'s tool discovery endpoint\n815\t\n816\t#### Tools not executing\n817\t\n818\t**Symptoms:** Tools are discovered but fail during execution\n819\t\n820\t**Troubleshooting:**\n821\t\n822\t1. **Parameter validation:** Ensure your tool accepts the expected parameters\n823\t2. **Schema compatibility:** Verify your input schemas are valid JSON Schema\n824\t3. **Error handling:** Check if your tool is throwing unhandled exceptions\n825\t4. **Timeout issues:** Consider increasing the `timeout` setting\n826\t\n827\t#### Sandbox compatibility\n828\t\n829\t**Symptoms:** MCP servers fail when sandboxing
